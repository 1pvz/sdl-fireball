## 编译 SDL

1. 从 github clone SDL 代码
```
git clone https://github.com/libsdl-org/SDL
```
clone 成功后会出现 `SDL` 目录

2. 编译 SDL 文件
先进入 `SDL` 目录
```
cd SDL
```
可以在目录下找到 `CMakeLists.txt` 文件，这个是 CMake 的配置文件

运行命令
```
cmake -S . -B build
```
- `-S` 指定源码目录（Source directory），也就是 `CMakeLists.txt` 所在的位置
  - `.` 表示当前目录
- `-B` 指定构建目录（Build directory），也就是生成的构建文件放在哪个目录
  - `build` 是目录名

如果没有报错，那么命令运行成功后会在 `SDL` 目录下出现 `build` 目录，里面有 build 环境相关的文件，点进去通常能看到不少 `.cmake` 文件

接着运行命令
```
cmake --build build
```
- `--build build` 指定从 `build` 目录开始构建

构建成功后会在 `build` 目录下生成 SDL 动态库文件 `libSDL3.0.dylib`

## 配置开发环境
接下来需要把 `SDL/include/SDL3` 目录下的所有头文件和 `SDL/build/libSDL3.0.dylib` 动态库文件复制到项目中，按照习惯，头文件会放在 `include` 目录，库文件会放在 `lib` 目录，也就是看起来是这种格式

``` shell
.
├── include
│   └── SDL3
│       ├── SDL_assert.h
│       ├── SDL_asyncio.h
│       ├── SDL_atomic.h
│       ├── ...
│       ├── 省略其他头文件
│       ├── ...
│       └── SDL.h
├── lib
│   └── libSDL3.0.dylib
└── main.c
```

其中 `main.c` 的内容很简单
``` c
#include <SDL3/SDL.h>
#include <SDL3/SDL_main.h>

int
main(int argc, char* argv[]) {
    return 0;
}
```

如果直接运行编译命令一般会报错
``` shell
clang main.c
```

提示找不到头文件
``` shell
main.c:1:10: fatal error: 'SDL3/SDL.h' file not found
    1 | #include <SDL3/SDL.h>
      |          ^~~~~~~~~~~~
1 error generated.
```

这是因为编译器会去系统默认目录查找头文件，而我们现在是把 `SDL` 头文件放在了 `include` 目录，所以需要指定 `include` 目录让编译器去这里查找

``` shell
clang main.c -Iinclude
```
- `-Iinclude` 指定头文件查找目录为 `include`

编译成功后会在当前目录生成可执行文件 `a.out`，可执行文件就是通常所说的程序，直接运行程序
``` shell
./a.out
```
因为现在这个程序其实啥也没执行，只是用来测试 `SDL` 头文件能不能找到，所以看不到结果很正常，到目前为止只使用了头文件，但没有使用 `dylib` 文件，接下来需要看看 `dylib` 文件怎么使用

在前面的基础上增加 `SDL_Init` 初始化函数

``` c
#include <SDL3/SDL.h>
#include <SDL3/SDL_main.h>

int
main(int argc, char* argv[]) {
    SDL_Init(SDL_INIT_VIDEO);
    return 0;
}
```

这次用同样的命令编译
``` shell
clang main.c -Iinclude
```

但这次有新报错
``` shell
Undefined symbols for architecture arm64:
  "_SDL_Init", referenced from:
      _main in main-be605f.o
ld: symbol(s) not found for architecture arm64
clang: error: linker command failed with exit code 1 (use -v to see invocation)
```

报错信息是指找不到 `_SDL_Init` 相关符号，符号在这里就是实现，头文件描述了函数长什么样子，比如接收什么参数，返回什么类型，但具体函数怎么实现的没在头文件提供，现在调用这些函数就报错了，而 `dylib` 包含函数的具体实现，所以还需要告诉编译器（链接器，也就是报错信息里的 `rd`）找到对应的 `dylib` 文件
``` shell
clang main.c -Iinclude -Llib -lSDL3.0
```
- `-Llib` 告诉链接器去 `lib` 目录找库
- `-lSDL3.0` 告诉链接器找 `SDL3.0` 库

合起来就是去 `lib` 目录下找 `SDL3.0` 库，但 `lib` 目录里的库文件名称是 `libSDL3.0.dylib`，而不是 `SDL3.0`，为什么也可以找到？

实际上库有一套约定的命名，即 `lib` + 库名称 + 库后缀名，`.dylib` 就是其中一种后缀名，所以通过 `-L` 和 `-l` 参数就可以找到对应的库

此时如果运行程序 `a.out` 会发现依然有报错
``` shell
dyld[37662]: Library not loaded: @rpath/libSDL3.0.dylib
  Referenced from: <D897BB93-88AD-3B40-841E-A1AAEFDEDC96> a.out
  Reason: no LC_RPATH's found
zsh: abort      ./a.out
```
这是因为 `dylib` 是动态运行库，也就是运行的时候才会去找对应的库文件，而链接的时候没有指定库文件，导致运行的时候找不到，就报错了

临时解决方案是通过环境变量的形式来提供动态库的搜索路径
``` shell
DYLD_LIBRARY_PATH=./lib ./a.out
```

但这样运行很不方便，所以更好的做法是在链接的时候就指定对应的库搜索路径

``` shell
clang main.c -Iinclude -Llib -lSDL3.0 -Wl,-rpath,./lib
```
这里的参数 `-Wl` 非常奇怪，需要特殊说明
- `-Wl,-rpath,./lib` 是一个整体，意思是把 `-rpath,./lib` 作为参数传递给链接器

当然这是兼容性写法，现在版本的 `Clang` 不需要这么麻烦，可以直接写成比较常规的形式
``` shell
clang main.c -Iinclude -Llib -lSDL3.0 -rpath ./lib
```
- `-rpath ./lib` 指定去 `lib` 目录找库

## 最小程序
前面是通过程序没有报错的形式来验证程序是否成功，这种方式并不方便，所以接下来创建一个窗口，更直观地演示

``` c
#include <SDL3/SDL.h>
#include <SDL3/SDL_main.h>

int
main(int argc, char* argv[]) {
    if (SDL_Init(SDL_INIT_VIDEO) == false) {
        SDL_Log("Error initializing SDL: %s\n", SDL_GetError());
        return 1;
    }

    const int w = 640;
    const int h = 480;
    SDL_Window *window = SDL_CreateWindow("Hello SDL3", w, h, 0);
    if (window == NULL) {
        SDL_Log("sdl create window failed %s\n", SDL_GetError());
        return 1;
    }

    SDL_Surface *surface = SDL_GetWindowSurface(window);
    if (surface == NULL) {
        SDL_Log("Error getting surface: %s\n", SDL_GetError());
        return 1;
    }

    Uint32 color = SDL_MapSurfaceRGB(surface, 255, 0, 0);
    SDL_FillSurfaceRect(surface, NULL, color);
    SDL_UpdateWindowSurface(window);

    SDL_Event event;
    while (SDL_WaitEvent(&event)) {
        if (event.type == SDL_EVENT_QUIT) break;
    }

    SDL_DestroyWindow(window);
    window = NULL;
    surface = NULL;
    SDL_Quit();

    return 0;
}
```
运行成功后会创建一个 640 * 480 且带有标题的窗口，背景是红色

### 初始化 SDL
程序会在起点的位置初始化 SDL，如果初始化过程失败，输出对应的报错信息，`SDL_Log` 和 `SDL_GetError` 通常会固定出现，用来输出 `SDL` 环境报错信息
``` c
if (SDL_Init(SDL_INIT_VIDEO) == false) {
    SDL_Log("Error initializing SDL: %s\n", SDL_GetError());
    return 1;
}
```

### 创建窗口
`SDL3` 简化了创建窗口的 API，通常指定标题、宽、高即可，第四个参数是指窗口的类似于，0 表示默认窗口，还有其他各种值，比如 1 表示全屏窗口
``` c
const int w = 640;
const int h = 480;
SDL_Window *window = SDL_CreateWindow("Hello SDL3", w, h, 0);
if (window == NULL) {
    SDL_Log("sdl create window failed %s\n", SDL_GetError());
    return 1;
}
```

### surface 表面
创建窗口之后需要把它画出来，`SDL` 提供了 `surface` 来做这个事情，相当于提供一块画布，把想要的内容画在画布上。创建 `surface` 也可能会失败，所以依然需要加一层判断

``` c
SDL_Surface *surface = SDL_GetWindowSurface(window);
if (surface == NULL) {
    SDL_Log("Error getting surface: %s\n", SDL_GetError());
    return 1;
}
```

有了 `surface` 之后，画一个红色实心矩形来测试是否生效，`SDL_MapSurfaceRGB` 把 rgb 颜色转成 Uint32 类型，然后调用 `SDL_FillSurfaceRect` 画实心颜色，接着 `SDL_UpdateWindowSurface` 更新 `surface` 颜色
``` c
Uint32 color = SDL_MapSurfaceRGB(surface, 255, 0, 0);
SDL_FillSurfaceRect(surface, NULL, color);
SDL_UpdateWindowSurface(window);
```

Mac 需要特殊处理，不然不会把窗口画出来，所以加一个循环告诉 Mac 直接画出窗口
``` c
SDL_Event event;
while (SDL_WaitEvent(&event)) {
    if (event.type == SDL_EVENT_QUIT) {
        break;
    }
}
```

### 回收资源
最后在程序结束之前，销毁相关变量，回收资源
``` c
SDL_DestroyWindow(window);
window = NULL;
surface = NULL;
SDL_Quit();
```
